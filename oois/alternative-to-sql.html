Alternatives to SQL

Initial post

Over the last decade, there has been an increase in demand for applications which must deal with vast quantities of unstructured data. To meet demand, there is a significant trend toward more relaxed databases in the form of NoSQL databases (Alotaibi & Pardede, 2019). NoSQL databases use many data models to access and manage data, and their best feature is that most of them are open-source. These databases are primarily intended for applications that demand significant data volumes, low latency, and flexible data models obtained by loosening specific classic databases' data consistency limitations.

There are several NoSQL databases, the most popular ones being document databases, graph databases, key-value databases and wide-column databases. Each has specific benefits and is utilized based on the use case at hand.

Document databases, which are sometimes referred to as document stores, serve best for storing partially structured data in a document format. Their popularity grew with the rise of JavaScript usage.

Graph databases, also called graph stores, are similar to classical RDS in the sense that they store data in nodes akin to rows in relational databases.

Key-value databases have basic data models that correlate a unique key with a value, which can be perfectly utilized in caching or managing limited sessions in apps.

Like an RDBMS, wide-column stores use rows, columns, and tables to store data, but unlike relational system, the columns can vary from row to row in the same table in a wide-column store. The column names can also be changed as needed, which provides a level of schema flexibility that does not exist in an RDBMs (Datanami, 2021).

Peer Response to Rachel Doherty

Hi Rachel,
Thank you for a very insightful essay!

The term "NoSQL" (short for "not just SQL") has recently come to refer to a broad spectrum of databases that lack the characteristics of traditional relational databases and are not frequently searched with SQL (structured query language).

It is essentially a vast serialized object store that allows one to retrieve objects by ID. It generally lacks an organized schema and does not handle complex queries. It encourages denormalization and is built from the ground to be distributed (cloud-scale). As a result, the ACID criteria are no longer required. Any database may answer any query, and any write query can act on any database, ultimately propagating to other distributed servers. Any server may respond to any query thanks to the distributed paradigm. Behind the scenes, servers speak with one another at their own pace. The server that responds to one's query might not have the most up-to-date information. As you have mentioned, NoSQL databases have a great application with the social media platforms but are still not popular enough for the large majority of corporate entities.

Peer Response to Nicolas Haas

Hi Nicolas,

thank you for your contribution - I found it quite insightful.

Utilizing NoSQL databases can definitely allow for more flexibility, especially when we have use cases such as the paragraph example you mentioned. I find that the aforementioned flexibility can have unforeseen costs later. For example the lack of consistency will force developers to write additional code for handling missing values, which would otherwise be handling direclty in the database. If we work with databases, which store data in key-value format, we are risking data inconsistency for certain keys, or rather columns (in relational db sense). All constraints would have to be handled directly in the application logic, rather than in the SQL code, within the database itself.

One benefit that traditional relational databases have is indexing. Indexes are a powerful tool used in the background of a database to speed up querying. Indexes power queries by providing a method to quickly lookup the requested data (Codecademy, n.d.).

To summarize, it can be concluded that there are two main benefits to NoSQL databases over traditional relational databases. First benefit is the flexibility that the NoSQL databases/stores offer. The second benefit is the cost. NoSQL databases are generally economical, i.e. cost-effective and can even be used for free, whereas the cost for 100 GB / month for cloud-based RDMS systems vs. NoSQL solutions is around 50 times greater than for the same amount of non-relational storage (lynnlangit, 2011).

Summary Post

The possibilities for document database applications are nearly endless. Consider NoSQL as an ERP option (Enterprise Resource Planning). The typical relational ERP database has begun to look like a piece of history, with PDFs, image files, audio files, metadata, and more now deemed critical corporate understanding.

Depending on the NoSQL database, JSON documents, graphs with nodes and edges, or key-value pairs can represent data.
Web apps which use RESTful APIs or need real-time data can be easily constructed by utilizing the advantages of the NoSQL databases.

These databases offer flexibility for data that has not been normalized, needs a flexible data model, or has various features for distinct data entities. In addition, they provide scalability for larger data sets.
Key-value stores are employed when a customer demands outstanding performance for serving queries on read-only data, such as for big, heavily trafficked websites (Datanami, 2021). For key-value storage, there is no query language; instead, developers use conventional CRUD commands.

While NoSQL databases have many benefits, they also come with some drawbacks. They are designed for specific use cases, while SQL databases are more suitable for general-purpose use cases.

Due to the eventual consistency model used by almost all the NoSQL databases, they can not provide the same level of consistent data as SQL databases. Data can sometimes be inconsistent, making them unsuitable for instantaneous integrity activities, such as banking or ATM transactions (SearchDataManagement, n.d.).
Another significant issue with NoSQL databases is the lack of industry standards compared to their relational, well-established counterpart. This is, of course, due to their novelty.

Relational databases will be around for many years, but NoSQL databases today are giving their relational cousins a run for their money, which is a great benefits to users (Datanami, 2021).
